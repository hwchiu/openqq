安裝

如何建立 Azure Kata 驗證環境

這次驗證使用專用 Terraform stack 與 wrapper script，在 Azure 上建立 3 台 VM、安裝 K3s 與 CRI-O、把 Kata 佈署到每個節點，最後再跑 runtime verify。Filesystem / CSI 驗證則是在這條 base path 成功後再額外執行；目前預設入口會跑 Azure Files SMB dynamic StorageClass。

必要檔案

這條安裝路徑會用到哪些 repo 檔案

以下是目前這條 Kata Azure 驗證路徑實際使用的入口點。

  • Stack： terraform/stacks/k3s-kata-134/main.tf
  • Stack variables： terraform/stacks/k3s-kata-134/variables.tf
  • Stack 範例值： terraform/stacks/k3s-kata-134/stack.auto.tfvars.example
  • One-shot installer： scripts/install-k3s-kata-134.sh
  • Prerequisite check： scripts/check-kata-prereqs.sh
  • Kata install logic： scripts/install-kata.sh
  • Filesystem / CSI 驗證 wrapper： scripts/verify-azurefile-csi.sh
  • Filesystem / CSI 預設腳本： scripts/tests/kata-azurefile-csi-smb.sh
  • Filesystem / CSI NFS 補充腳本： scripts/tests/kata-azurefile-csi-nfs.sh
前置準備

先準備 Azure 變數

最簡單的做法是使用共用 Terraform variables 檔。

cp terraform/stacks/common.auto.tfvars.example terraform/stacks/common.auto.tfvars
cp terraform/stacks/k3s-kata-134/stack.auto.tfvars.example terraform/stacks/k3s-kata-134/stack.auto.tfvars
subscription_id = "00000000-0000-0000-0000-000000000000"
tenant_id       = "00000000-0000-0000-0000-000000000000"
admin_username  = "azureuser"
ssh_public_key  = "ssh-ed25519 AAAA..."
One-shot

直接跑完整安裝流程

如果沒有特別 override，wrapper 會使用這次驗證時的預設 K3s 與 CRI-O 版本。

bash scripts/install-k3s-kata-134.sh
  • K3s 預設版本：v1.34.1+k3s1
  • CRI-O 預設 stream：v1.34
  • Stack 內 VM 規格：Standard_D4s_v3
  • Kata install script 預設版本：3.31.0
流程說明

Wrapper 實際做了哪些事

這個 wrapper 會固定依序跑完所有必要步驟。

  1. 對 terraform/stacks/k3s-kata-134 執行 Azure stack provisioning。
  2. 把 kubeconfig 抓到 generated/stacks/k3s-kata-134/kubeconfig。
  3. 等待 3 個節點都進入 Ready。
  4. 執行 scripts/check-kata-prereqs.sh，確認硬體虛擬化與 /dev/kvm。
  5. 在 control plane 與兩個 worker 上執行 scripts/install-kata.sh。
  6. 套用 k8s/kata-runtimeclass.yaml。
  7. 執行 scripts/verify-kata-runtime.sh 啟動 Kata-backed probe pod。
手動分段

如果只想重跑部分步驟

下面這組指令適合用來單獨重跑 prerequisite、install 或 verify。

KUBECONFIG_PATH=generated/stacks/k3s-kata-134/kubeconfig \
TF_DIR=terraform/stacks/k3s-kata-134 \
bash scripts/check-kata-prereqs.sh

KUBECONFIG_PATH=generated/stacks/k3s-kata-134/kubeconfig \
TF_DIR=terraform/stacks/k3s-kata-134 \
bash scripts/install-kata.sh

KUBECONFIG_PATH=generated/stacks/k3s-kata-134/kubeconfig \
bash scripts/verify-kata-runtime.sh
Filesystem / CSI

如果還要驗證 Azure Files CSI StorageClass

這不是 base install 的必要步驟，但如果你要重現這次 filesystem mount 驗證，需要再準備 Azure CLI、Helm 與可登入節點的 SSH 金鑰。預設入口會執行 Azure Files SMB dynamic StorageClass；如果要重跑 NFS 補充實驗，則再切換 profile。

  • 先完成本頁前半段的 3 VM、K3s、CRI-O、Kata 安裝。
  • 本機需要已完成 az login。
  • 本機需要可用的 helm。
  • 本機需要可 SSH 到 worker 的私鑰；預設讀取 ~/.ssh/id_ed25519，也可改用 AZURE_SSH_PRIVATE_KEY_PATH。
KUBECONFIG_PATH=generated/stacks/k3s-kata-134/kubeconfig \
TF_DIR=terraform/stacks/k3s-kata-134 \
bash scripts/verify-azurefile-csi.sh

KUBECONFIG_PATH=generated/stacks/k3s-kata-134/kubeconfig \
TF_DIR=terraform/stacks/k3s-kata-134 \
AZUREFILE_CSI_PROFILE=nfs \
bash scripts/verify-azurefile-csi.sh

預設腳本會先安裝 Azure File CSI driver 1.35.4，建立暫時的 Standard_LRS + StorageV2 storage account，透過 secret-backed dynamic StorageClass 自動 provision volume，驗證兩個 Kata pod 的跨節點共享掛載， 最後把 namespace 與 storage account 清掉。若切到 AZUREFILE_CSI_PROFILE=nfs， 則會改跑 Premium FileStorage + NFS share 的補充實驗。

CRI-O Wiring

Kata 會在每個節點改哪些 CRI-O 設定

install script 會先寫入兩個 CRI-O drop-in，再重新啟動相關服務。

[crio]
storage_option = [
  "overlay.skip_mount_home=true",
]

[crio.runtime.runtimes.kata]
runtime_path = "/usr/local/bin/containerd-shim-kata-v2"
runtime_type = "vm"
runtime_root = "/run/vc"
privileged_without_host_devices = true
這一頁描述的是目前已驗證的 Azure Kata 安裝路徑。
