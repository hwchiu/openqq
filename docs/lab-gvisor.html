gVisor 在堆疊中的位置

gVisor 插入在 containerd 與 Linux host kernel 之間，對 Pod 內所有 syscall 進行攔截與轉譯。OpenShell Supervisor 仍在 Pod 內運作，負責網路政策與 Landlock；gVisor 在更底層提供額外的核心隔離。

gVisor 核心元件

🧩
runsc（OCI Runtime）

取代 runc，作為容器的 OCI Runtime。k3s containerd config 新增 io.containerd.runsc.v1 handler。

🔵
Sentry（虛擬核心）

user-space 核心，攔截所有 syscall 並在自身的核心邏輯中處理，不讓 Pod 直接呼叫 host kernel。

📁
Gofer（I/O 代理）

負責 Sentry 與 host 檔案系統之間的 I/O 代理，進一步隔離 Pod 的檔案存取。

🏷️
RuntimeClass

k8s/gvisor-runtimeclass.yaml 定義 name: gvisor、handler: runsc，讓特定 Pod 指定使用 gVisor。

runc vs gVisor：隔離層比較

面向runc（實驗一）gVisor / runsc（實驗二）
syscall 處理直接傳遞到 host Linux kernel由 runsc Sentry 攔截、在 user-space 轉譯
kernel 暴露面完整 host kernel（cgroups + namespaces 隔離）Sentry 虛擬核心隔離，host kernel 僅暴露有限接口
Landlock（OpenShell）由 host kernel 執行 Landlock LSM由 Sentry 虛擬核心執行（行為待驗證）
netns 建立需要 privileged: true workaround（k3s 限制）同樣需要；gVisor 不改變 Supervisor 的 capability 需求
cgroup pids.maxk3s 環境下不可用同樣預期不可用（k3s 限制，非 gVisor 問題）
容器啟動速度快（直接 exec）稍慢（需初始化 Sentry + Gofer）
syscall 相容性完整 Linux ABISentry 實作子集，少數 syscall 可能行為不同
privileged 容器授予 host 完整特權Sentry 內執行，host 層面特權效果受限（核心實驗假設）

對 runc 環境使用者：換成 gVisor 會有什麼不同？

如果你已經在跑 OpenShell + runc（參見 實驗一報告）， 切換到 gVisor 後以下幾個面向會有明顯差異。 本節列出已知差異點，並標注哪些需要實驗驗證。

1. 需要額外部署：RuntimeClass + gVisor Patcher

runc 環境只需要部署 openshell-sandbox-patcher.yaml（patch privileged）。gVisor 環境需要多兩個步驟：

runc（實驗一）
  • 部署 openshell-sandbox-patcher.yaml
  • Patcher 只 patch privileged: true
gVisor（本實驗）
  • 節點層：make gvisor-install（安裝 runsc，patch containerd config）
  • 叢集層：k8s/gvisor-runtimeclass.yaml（RuntimeClass）
  • Patcher：openshell-sandbox-patcher-gvisor.yaml（patch privileged: true AND runtimeClassName: gvisor）
k8s/openshell-sandbox-patcher-gvisor.yaml（核心邏輯片段）
# 同時 patch privileged + runtimeClassName，兩者任一不符即刪 Pod 觸發重建
patch_priv='[{"op":"add","path":".../securityContext/privileged","value":true}]'
patch_rc='[{"op":"add","path":"/spec/runtimeClassName","value":"gvisor"}]'
kubectl patch sandbox "${name}" --type json -p="${patch_priv}"
kubectl patch sandbox "${name}" --type json -p="${patch_rc}"

if [ "$pod_runtime" != "gvisor" ] || [ "$pod_privileged" != "true" ]; then
  kubectl delete pod "${name}" --ignore-not-found --wait=false
fi

2. SecurityContext：需求不變，效果可能不同

OpenShell Supervisor 同樣需要 privileged: true、SYS_ADMIN、NET_ADMIN 等 capabilities。

影響項目runc（實驗一）gVisor（本實驗）
privileged: true 的意義 Pod 直接擁有 host 特權，可操作 host 資源 Sentry 內的虛擬核心允許特權操作，但不直接暴露 host（⚠️ 待驗證）
netns 建立（Supervisor） 需 privileged workaround 同樣需要 privileged；gVisor 本身不改善此限制
SYS_ADMIN / NET_ADMIN 直接作用在 host cgroup / netns 作用在 Sentry 虛擬核心；部分操作行為可能不同（⚠️ 待驗證）
Pod Security Admission 需 privileged namespace 同樣需要 privileged namespace

⚠️ 核心假設：gVisor 下的 privileged 容器是否仍能讓 Supervisor 正常建立 netns、執行 Landlock，是本實驗的關鍵驗證點。

3. 網路模型：Supervisor netns 路徑不變

OpenShell Supervisor 的網路隔離靠 Pod 內 veth + netns（而非 CNI NetworkPolicy），這個機制與 runc 或 gVisor 無關。流量路徑相同：

預期 Supervisor 日誌（與 runc 相同結構）
CONFIG:CREATING  Creating network namespace [ns:sandbox-xxxxxxxx host_veth:veth-h-xxxxxxxx sandbox_veth:veth-s-xxxxxxxx]
CONFIG:CREATED   Network namespace created [ns:sandbox-xxxxxxxx host_ip:10.200.0.1 sandbox_ip:10.200.0.2]
CONFIG:INSTALLED Bypass detection rules installed [ns:sandbox-xxxxxxxx]
NET:LISTEN       10.200.0.1:3128   # ← 相同 CONNECT proxy 結構
面向runcgVisor
CONNECT Proxy 位置10.200.0.1:3128預期相同（待確認）
L7 HTTP 政策管控✓預期 ✓（Supervisor 在 gVisor 內同樣管控）
NetworkPolicy 效果CNI 層管控，與 Supervisor 無關相同
bypass 偵測規則iptables 規則阻止繞過 Proxyiptables 在 Sentry 內執行，行為待驗證 ⚠️

4. Landlock：在 gVisor Sentry 內執行

runc 下 Landlock 由 host Linux kernel 直接執行（本 lab 已確認 ABI v4）。gVisor 下，Landlock 呼叫由 Sentry 的虛擬核心處理。

項目runc（已確認）gVisor（預期）
Landlock ABI 版本v4（host kernel 5.19+）依 Sentry 實作，可能不同 ⚠️ 待確認
/tmp 寫入允許✓ TMP_OK⏳ 待執行
/var/tmp 拒絕✓ VARTMP_DENIED⏳ 待執行
根目錄列舉拒絕✓ Permission denied⏳ 待執行
cgroup pids.max不可用（k3s 限制）預期同樣不可用

5. Observability：OCSF 日誌結構相同

gVisor 本身不改變 Supervisor 的 log 格式。OCSF 事件仍由 Supervisor stdout 輸出。但 runsc 本身有自己的診斷日誌，可與 Supervisor log 互補：

Log 來源runcgVisor
Supervisor OCSF 日誌kubectl logs <pod>相同
OCI Runtime 日誌（runc 無獨立日誌）runsc 可啟用 --debug，輸出 syscall 攔截詳情
節點層 containerd log標準 containerd 事件同樣來自 containerd，但 runtime 欄位顯示 runsc

6. 偵錯：多一層 gVisor Sentry

場景runc sandboxgVisor sandbox
kubectl exec 進入 Pod Supervisor 管理的環境 同樣由 Supervisor 管理；syscall 先過 Sentry
查看 syscall 行為 用 strace 在 Pod 內 Sentry 攔截 strace 本身；建議從 runsc --debug 取得 syscall 追蹤
Pod 起不來 確認 privileged patch + CONFIG:CREATING log 額外確認：kubectl get pod -o yaml 中 runtimeClassName: gvisor 是否已 patch；runsc 是否在節點上可用
RuntimeClass 未就緒 N/A Pod 停在 Pending，events 顯示 no nodes available to schedule pods — 需確認 make gvisor-install 已完成

7. 已知 gVisor 限制（實驗前預知）

限制影響嚴重性
部分 syscall 未實作 特殊 syscall 呼叫可能回傳 ENOSYS；常見 AI agent 工具鏈（curl, python3）通常無問題 低（需個案評估）
效能開銷 syscall 攔截帶來約 2–10x latency（依 workload 不同），I/O 密集型 agent 較明顯 中（視 workload）
privileged 容器隔離語意模糊 gVisor 對 privileged 的支援不完整；Supervisor 所需能力是否全部可用尚未確認 高（核心驗證點）
Landlock + Sentry 相容性 gVisor Sentry 的 Landlock 實作可能與 host kernel 行為不同 中（需實驗驗證）
k3s containerd config template 需手動維護 k3s 升級後可能覆蓋 config.toml.tmpl，需重新 patch（make gvisor-install 可重跑） 低（維運注意）

如何重現 gVisor 實驗

以下步驟從已有 runc 叢集出發，在現有叢集加裝 gVisor（無需重建 VM）。

1
確認叢集已就緒（runc 實驗完成後） 
$ make cluster-status
# 確認 3 節點均 Ready
$ kubectl get nodes -o wide
2
在每個 worker 安裝 gVisor + 設定 containerd 
$ make gvisor-install
# scripts/install-gvisor.sh：
# → SSH 進 worker-1, worker-2
# → apt install runsc
# → patch /var/lib/rancher/k3s/agent/etc/containerd/config.toml.tmpl
# → systemctl restart k3s-agent
# → kubectl apply -f k8s/gvisor-runtimeclass.yaml
3
驗證 RuntimeClass 可用 
$ make gvisor-verify
# 起一個 runtimeClassName: gvisor 的探針 Pod
# 成功時輸出 uname -r 顯示 gVisor 虛擬核心版本（例如：4.4.0）
# 而非 host kernel 版本（例如：5.15.0-...）
4
部署 gVisor 版 Sandbox Patcher 
$ make openshell-patcher-gvisor
# 部署 openshell-sandbox-patcher-gvisor Deployment
# Patcher 監聽 Sandbox CR，同時 patch privileged=true + runtimeClassName=gvisor
5
確認 OpenShell 仍在運行 
$ kubectl get pods -n openshell
# Gateway + Controller 均 Running
$ openshell endpoint list
6
執行 7 項驗證實驗 
$ make openshell-verify
# 結果儲存至 testing/raw/verify-gvisor-<timestamp>/
# 對比 testing/raw/verify-<runc-timestamp>/ 找出差異
7
清理資源 
$ make cluster-down
# 銷毀所有 Azure 資源，避免計費

全新叢集直接使用 gVisor（cloud-init 路徑）

若要從頭建立一個使用 gVisor 的叢集（不經過 runc 階段），可設定 Terraform 變數：

terraform/terraform.tfvars
container_runtime = "gvisor"
# cloud-init 會自動執行：
# 1. INSTALL_K3S_SKIP_START=true 安裝 k3s
# 2. 安裝 runsc apt 套件
# 3. 設定 containerd config.toml.tmpl
# 4. 啟動 k3s / k3s-agent
建立叢集
$ make cluster-up
# 叢集建立後無需手動執行 make gvisor-install
$ make gvisor-verify  # 仍需驗證 RuntimeClass
$ make openshell-install && make openshell-patcher-gvisor
