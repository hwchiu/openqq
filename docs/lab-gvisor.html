GVISOR TRADE-OFF Isolation branch

gVisor is not an upgrade,
it is a trade-off

This line is not a failure case. It is a controlled regression case. OpenShell keeps L7 policy, binary allowlist, and hot-reload on runsc, but loses filesystem policy because the Landlock syscall path is unavailable.

Full gVisor report Landlock root cause
Most important finding

This is not a policy-distribution problem. The actual root cause is `landlock_create_ruleset` returning `ENOSYS` on the gVisor path, which means the supervisor cannot activate filesystem rules.

WHAT SURVIVES

Capabilities that stay alive on runsc

L7 policy remains

`curl GET` is allowed, `curl POST` is denied, and `python3 GET` is denied. The network/L7 policy plane still works across the runtime boundary.

Hot-reload remains

Matching Pod UIDs across a policy update show that the OpenShell control plane can still update active policy without a pod rebuild.

The sandbox still starts

Unlike the Kata blocker path, this runtime reaches real OpenShell behavior validation. The failure is inside one enforcement layer, not at startup.

WHAT BREAKS

The specific regression layer

Filesystem policy is the failed layer. The supervisor reports `Landlock Filesystem Sandbox Unavailable`, and the failure can be reproduced through direct syscall comparison rather than inferred from policy behavior alone.
Probe runc gVisor Meaning
`/var/tmp` write Denied Allowed Filesystem policy no longer blocks write access on gVisor.
`ls /` Denied Allowed Root-directory listing is no longer constrained by Landlock.
Landlock syscall ret=4 ENOSYS The root cause is concrete and directly measurable.
