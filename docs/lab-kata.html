OpenShell + Kata
相容性報告

這條線不是失敗於「Kata 沒跑起來」，而是失敗於 OpenShell 目前需要的 privileged sandbox 與 Kata 的相容性。
因此這頁的重點不是功能展示，而是阻塞點定位。

目前真實狀態

Azure 叢集已切換到 k3s + containerd + RuntimeClass kata，且 `kata-verify` probe 已成功回報 Kata guest kernel。阻塞點是 OpenShell sandbox 在被 patch 成 privileged: true 與 runtimeClassName: kata 後，Kata 會回 QMP command failed: The device is not writable: Permission denied。

  • `kata-alpine-priv-imagevol` 測試
  • 真實的 OpenShell sandbox `kata-proof`

為什麼這對 OpenShell 很重要

在這個 lab 的 `k3s` 路徑下，OpenShell sandbox 並不是一個普通的 unprivileged pod。它仍然需要：

  • privileged: true
  • SYS_ADMIN、NET_ADMIN 等額外 capability
  • Supervisor 建立 netns 與 proxy 的能力

因此，雖然 Kata runtime 可跑、OpenShell control plane 也可跑，但只要 sandbox 這一層仍依賴 `privileged`，目前就無法把 OpenShell 真正落到 Kata 上。

官方文件交叉驗證

官方訊號我們的實驗怎麼對上
OpenShell Security Best Practices 明確寫出 supervisor 啟動順序仍包含 privileged bootstrap helpers 與 netns setup這解釋了為什麼單純開 user namespaces 不是現成解法，因為它沒有取消 bootstrap 階段對高權限的依賴。
Gateway config 允許 server.enableUserNamespaces=true這是 defense-in-depth 開關，不是「讓 Kubernetes driver 變成無 privileged」的保證。
Gateway config 允許 supervisor.sideloadMethod=init-container我們已用最小矩陣證明 image volume 不是主因，所以切 sideload method 不會解決目前的 Kata 阻塞。
OpenShell 官方另有 vm compute driver如果目標是要 VM 邊界，官方路徑其實是 VM driver，而不是把 Kubernetes sandbox 硬套進 Kata。

這一節的重點是收斂方向：user namespaces 是加分項，不是 privileged 問題的替代解；如果要追 OpenShell 官方支持的 VM 邊界，下一條更合理的路是直接驗證 `compute_drivers = [\"vm\"]`。

為什麼我沒有直接改成 User Namespaces 測試

原因很直接：目前官方文件自己就說明了 supervisor 啟動順序仍依賴 privileged bootstrap helpers 與 `setns`。這表示 user namespaces 最多只能降低 host 暴露面，不能直接消掉 OpenShell 目前在 Kubernetes driver 下的高權限啟動需求。

實驗決策
保留現有 Kata 實驗證據
不破壞目前可工作的 Azure / OpenShell / Kata runtime 狀態
不把 user namespaces 誤寫成 privileged 替代方案

更合理的下一條路

Path A

追 OpenShell VM Driver

官方本來就有 `vm` compute driver，文件寫明它提供 VM-backed sandbox。若目標是「OpenShell + VM 邊界」，這條路比 Kubernetes+Kata 更接近官方設計。

Path B

追無 privileged sandbox

如果未來 OpenShell 的 Kubernetes driver 能去掉 privileged bootstrap helpers，Kata 才有機會變成真正可行的第三條正式路徑。

Path C

把 Kata 留在對照組

現在最務實的定位，是把 Kata 保留成 runtime 對照組與 blocker 證據，不把它包裝成已可上線的 OpenShell 部署選項。

建議解讀

What Works

Kata 可當 runtime 對照組

Kata 這條線已經足夠拿來跟 runc / gVisor 比較 runtime 行為與 VM 邊界，不需要再懷疑 Azure 規格或 containerd 整合。

What Blocks

OpenShell sandbox 目前被 privileged 卡住

阻塞點不在 Helm、不在 patcher、不在 RuntimeClass，而在 OpenShell sandbox 的權限需求與 Kata 的相容性。

Next Step

要追的是權限模型，不是一般 policy

若要讓 Kata 成為正式選項，下一步應釐清 OpenShell 是否能移除 `privileged` 依賴，或改用另一條支援路徑。
