INTERPRETATION

What the blocker actually means

This does not prove that Kata itself is broken. It proves that the current OpenShell sandbox shape in this k3s path still depends on a privileged pod configuration that Kata rejects in this environment.

  • The runtime is installed correctly and can execute Kata-native workloads.
  • The patcher is injecting `runtimeClassName: kata` and `securityContext.privileged: true` as intended.
  • The failure occurs after that injection, at the point where Kata refuses the privileged container shape.
NEXT MOVES

What would unblock progress

Any forward path has to address the privileged dependency directly rather than rewriting ordinary policy YAML.

  • Determine whether OpenShell can remove the privileged bootstrap requirement on this path.
  • Verify Kata support boundaries for privileged workloads and any viable alternative settings.
  • If the gap remains, treat Kata as a runtime comparison line rather than the main OpenShell deployment path.
