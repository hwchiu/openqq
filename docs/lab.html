對 Kubernetes 原生使用者：導入 OpenShell 會有什麼不同？

如果你習慣用 kubectl apply、NetworkPolicy、PodSecurityAdmission 管理工作負載， 導入 OpenShell 後有幾個面向會和你的既有經驗產生明顯落差。 以下依影響程度列出，並附上本 lab 的實際證據。

1. 雙控制平面：kubectl 之外多了一個 openshell CLI

k8s 原生的操作習慣是「一切都是 YAML，一切都走 API Server」。引入 OpenShell 後：

kubectl（仍然存在）
  • 查看 Sandbox CR / Pod / Service
  • 讀 container logs（kubectl logs）
  • 存取 k8s RBAC、ConfigMap 等

但你不該直接用 kubectl 刪 Sandbox——應該走 OpenShell CLI 確保 Gateway 狀態一致。

openshell CLI（新增）
  • openshell sandbox create / delete / list
  • openshell policy apply / get
  • openshell sandbox exec（進入受保護 shell）

Gateway 維護 sandbox session 狀態；繞過 CLI 直接改 k8s 資源會讓 Gateway 與 k8s 進入不一致。

⚠️ 這意味著你的 GitOps 流程（ArgoCD / Flux）需要重新思考：
哪些資源走 kubectl apply？哪些要走 OpenShell CLI？
OpenShell 自己建立的 Sandbox CR 帶有 openshell.ai/managed-by: openshell annotation， 不應被 GitOps 工具覆蓋。

2. SecurityContext 的現實衝擊

這是影響最大、也最容易踩雷的地方。OpenShell Supervisor 需要在 Pod 內建立 network namespace（veth pair、iptables），這要求較高的 Linux capabilities。

本 lab 實際觀測到的 securityContext：

testing/raw/architecture-evidence/proof-auto-sandbox.yaml（securityContext 節錄）
securityContext:
  privileged: true
  runAsUser: 0
  capabilities:
    add:
      - SYS_ADMIN
      - NET_ADMIN
      - SYS_PTRACE
      - SYSLOG
影響項目說明本 lab 狀態
Pod Security Admission（PSA） privileged: true 在 restricted / baseline namespace 下會被拒絕，只有 privileged namespace 才放行 openshell namespace 需設為 privileged
OPA / Gatekeeper / Kyverno 常見 policy 禁止 privileged: true 或 runAsUser: 0，OpenShell sandbox 會觸發這些規則 需要明確加入 exemption
雲端 managed k8s（EKS / GKE / AKS） 部分平台預設開啟 PSA 或節點層級安全加固，privileged Pod 可能需要額外 node pool 設定 未測試
k3s（本 lab） containerd 預設不支援 Pod 內 mount namespace 操作，netns 建立失敗。需要 privileged workaround 已解決（patcher）

k3s 的特殊 workaround：sandbox-patcher

因為 OpenShell 建立 Sandbox CR 時不會預設加上 privileged: true，本 lab 需要一個 sidecar 腳本持續 patch：

k8s/openshell-sandbox-patcher.yaml（核心邏輯）
# 持續監看 Sandbox CR，一旦 privileged != true 就 patch
patch='[{"op":"add","path":"/spec/podTemplate/spec/containers/0/securityContext/privileged","value":true}]'
kubectl patch sandbox "${name}" --type json -p="${patch}"

# 如果 Pod 已建立但沒有 privileged，刪掉讓 controller 重建
kubectl delete pod "${name}" --ignore-not-found --wait=false

💡 這個 patcher 是 lab 環境的 workaround，不是 OpenShell 正式提供的元件。在正式環境中應由 MutatingWebhook 或節點配置解決。

3. 網路模型：NetworkPolicy 在這裡不管用

k8s 原生的出口控制通常靠 NetworkPolicy（CNI 實作）。OpenShell 的網路隔離走完全不同的路：

Supervisor 啟動時的 netns 建立（來自 sandbox-logs.txt）
CONFIG:CREATING  Creating network namespace [ns:sandbox-8dbef1f1 host_veth:veth-h-8dbef1f1 sandbox_veth:veth-s-8dbef1f1]
CONFIG:CREATED   Network namespace created [ns:sandbox-8dbef1f1 host_ip:10.200.0.1 sandbox_ip:10.200.0.2]
CONFIG:INSTALLED Bypass detection rules installed [ns:sandbox-8dbef1f1]
NET:LISTEN       10.200.0.1:3128   # ← Supervisor 自己起的 CONNECT proxy
能力k8s NetworkPolicyOpenShell Supervisor
作用範圍Pod 到 Pod 之間（東西向）Pod 內 agent 到外部（南北向 egress）
隔離機制CNI（iptables / eBPF）Pod 內獨立 Linux network namespace + veth
L7 感知（HTTP method/path）✗✓ CONNECT Proxy
預設 deny all egress需要明確寫 policy預設行為，不寫 = 全封鎖
變更方式kubectl apply NetworkPolicy YAMLopenshell policy apply（熱重載）
與 Calico / Cilium 的關係CNI 本身的 feature互不影響，但兩套規則都會生效

⚠️ 如果你同時有 NetworkPolicy + OpenShell policy，兩者都要放行流量才能連通。 NetworkPolicy 在 Pod 層控制進出，OpenShell 在 Pod 內的 netns 層控制 agent 的行為。

4. 雙重認證：k8s SA Token + OpenShell JWT

k8s 原生的 Pod 認證靠 ServiceAccount token（自動掛載）。OpenShell 多了一層自己的 JWT：

Supervisor 啟動認證流程（來自 sandbox-logs.txt）
Fetching sandbox policy via gRPC endpoint=http://openshell.openshell.svc.cluster.local:8080
exchanging K8s ServiceAccount token for sandbox JWT endpoint=http://openshell.openshell.svc.cluster.local:8080

注意 Sandbox CR 中：

automountServiceAccountToken: false   # ← agent 的 SA token 預設關閉
volumeMounts:
  - mountPath: /var/run/secrets/openshell
    name: openshell-sa-token              # ← 只有 openshell bootstrap SA token
面向說明
agent 無法直接存取 k8s APIautomountServiceAccountToken: false，agent 拿不到 SA token，無法呼叫 k8s API Server
Supervisor 本身用 bootstrap SA 換 JWT這個 JWT 是 OpenShell Gateway 簽發，只用於 Supervisor ↔ Gateway 通訊
k8s RBAC 仍然需要openshell controller 本身仍需要 k8s RBAC 來建立 Pod / Service / Sandbox；需要獨立規劃 ServiceAccount + Role

5. Observability：OCSF 結構化日誌，不是一般 container log

kubectl logs <sandbox-pod> 輸出的是 OCSF 格式的結構化事件，不是你熟悉的應用 log：

kubectl logs proof-auto-1780439554（節錄）
[1780439796.224] [sandbox] [OCSF ] CONFIG:ENABLED  [INFO] TLS termination enabled: ephemeral CA generated
[1780439796.224] [sandbox] [OCSF ] CONFIG:CREATING [INFO] Creating network namespace [ns:sandbox-8dbef1f1 ...]
[1780439815.123] [sandbox] [OCSF ] NET:OPEN        [INFO] openshell.openshell.svc.cluster.local:8080
[1780439822.456] [sandbox] [OCSF ] HTTP:GET        ALLOWED  https://api.anthropic.com/v1/messages
[1780439823.789] [sandbox] [OCSF ] HTTP:POST       DENIED   https://evil.example.com/exfil
項目k8s 原生OpenShell
log 格式應用自定義（text / JSON）OCSF 固定結構：timestamp, category, severity, message
log 來源應用 stdout/stderrSupervisor stdout（包含所有安全事件）
agent 本身的 stdout直接可見混在 Supervisor log 裡，需要過濾 [PROC:] 事件
Prometheus metrics應用自己 expose目前版本未見 metrics endpoint（本 lab 未觀測到）
Event 類型N/ACONFIG / NET / HTTP / PROC / LIFECYCLE / SSH 等分類

💡 如果你有 ELK / Loki / Splunk，需要為 OCSF 格式加 parser。 [OCSF ] 開頭的行是安全審計事件，其餘是 Supervisor 自身的 operational log。

6. 偵錯（Debug）困難點

進入 sandbox 後的環境與一般 container 不同，以下是常見的混淆點：

場景一般 containerOpenShell sandbox
kubectl exec 進入 Pod 直接拿到 sh/bash 進入的是 Supervisor 管理的環境，binary allowlist 生效，curl 等可能不在白名單
查網路連線 ss -t / netstat 需在 sandbox netns 內查（ip netns exec sandbox-XXXX ss -t），從 host 上看
抓封包 tcpdump -i eth0 需抓 veth-h-XXXX 介面（host side），或在 sandbox netns 內抓 veth-s-XXXX
policy 不生效 N/A 確認 Supervisor 是否從 Gateway 拿到 policy（看 CONFIG:LOADED log），或退回 local disk policy
Pod 起不來 看 kubectl describe pod 先確認 privileged: true 是否已 patch；看 Supervisor 啟動階段的 CONFIG:CREATING / CONFIG:ENABLED

7. 已知限制與本 Lab 的特殊處理

k3s 環境的已知問題
Network namespace creation failed and proxy mode requires isolation
mount --make-shared /run/netns failed: Permission denied
→ 原因：k3s containerd 預設不允許 Pod 內做 mount namespace 操作
→ 解決：sandbox pod 設為 privileged: true（透過 patcher）

runtime cgroup pids.max is unavailable (No such file or directory)
→ 原因：k3s 的 cgroup 設定不完整，缺少 pids subsystem
→ 影響：無法對 agent child process 施加 PID 數量上限
→ 嚴重性：低，但表示 PID exhaustion 防禦缺失
問題影響正式環境建議
privileged workaround 安全邊界降低；違反 PSA restricted 在支援 netns 的節點上（如 Kata 之外的標準 k8s）應不需要；或用 MutatingWebhook 統一處理
cgroup pids.max 不可用 無 PID 上限防護 使用完整 cgroup v2 的節點（Ubuntu 22.04+ 預設）
policy local fallback Gateway 下線時仍使用 image 內的 policy.yaml 確保 image 內的 fallback policy 足夠保守（deny all 為佳）
TLS policy deprecation warning tls: terminate 已棄用，改自動 TLS 更新 policy 改用 tls: skip（若不需 TLS 終止）
