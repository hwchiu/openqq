目前真實狀態

Azure 測試環境已切換成 k3s + containerd + RuntimeClass gvisor。OpenShell 與 agent-sandbox 已重裝完成，gVisor probe Pod 成功回報 4.19.0-gvisor。OpenShell 的預設 deny egress、binary allowlist、L7 method/path 控制、policy hot-reload 在 gVisor 上仍成立；但 testing/raw/verify-1780497002/filesystem.txt 與 sandbox-logs.txt 也明確證明這組合下 Landlock Filesystem Sandbox Unavailable，所以 gVisor 組不是單純「更安全」，而是「底層隔離更強，但 OpenShell 的 filesystem enforcement 退化」。

決策摘要

現在應該怎麼選

  • 如果你要 OpenShell 的完整能力，尤其是 filesystem_policy，目前應優先選 `runc`。
  • 如果你要更強的 runtime / kernel 邊界，`gVisor` 有價值，但要接受目前 `Landlock` 缺口。
  • 目前最穩定的說法不是「gVisor 比 runc 更安全」，而是「兩者保護面不同」。

agent-sandbox 到底在解什麼問題？

agent-sandbox 是 Kubernetes SIG 的專案，目標是為「AI agent 工作負載」在 Kubernetes 上定義一個 標準化的 CRD 抽象，解決的是比 Pod 更高層的生命週期管理問題。

類比：StatefulSet 是為「有狀態服務」設計的 Pod 抽象；
agent-sandbox 的 Sandbox CRD 是為「AI agent 任務」設計的 Pod 抽象。
它本身不提供任何安全管控，只管「這個 agent workload 應該長什麼樣子、怎麼跑、怎麼存活」。

純 agent-sandbox 能給你什麼

下面是一個最小的 Sandbox CR，不需要 OpenShell，直接可以用：

k8s/agent-sandbox-smoke.yaml
apiVersion: agents.x-k8s.io/v1alpha1
kind: Sandbox
metadata:
  name: my-agent
spec:
  podTemplate:
    spec:
      containers:
        - name: agent
          image: busybox:1.36
          command: ["/bin/sh", "-c", "sleep infinity"]

套用後，controller 會自動幫你建好 Pod、Service、PVC，並維持 singleton 語意。

三個獨立維度：不要混在一起看

理解這個系統的關鍵是把三個維度拆開：

☸️
維度 1：工作負載管理

agent-sandbox CRD 解決的問題：「如何在 Kubernetes 上表示與管理一個 AI agent 生命週期？」

選項：直接用 Pod（無 CRD）
or 用 Sandbox CRD（推薦）

⚙️
維度 2：Container Runtime

runtimeClassName 解決的問題：「這個 Pod 要用什麼底層 runtime 執行？」

選項：runc（預設）/ Kata（VM）/ gVisor（user-space kernel）
與維度 1、3 正交，可自由組合

🔒

OpenShell Supervisor 解決的問題：「這個 agent 被允許做哪些事？」

選項：不裝（無管控）
or 裝 OpenShell（有 policy enforcement）

🛡️ 不用 Kata / gVisor 時，OpenShell 用什麼補？

以下是「⭐ 本 repo 組合：Sandbox CRD + runc + OpenShell」的實際防護層剖面圖。

🏠 Host OS（Linux Kernel + containerd）
⚠️ runc 共享 host kernel — 若 container escape 成功，攻擊者可直達 host
📦 Container（runc） — Linux namespace + cgroup 隔離
🔒 OpenShell Supervisor（PID 1）— 以下為 agent 執行前套用的所有策略
🗂 Landlock LSM

→ agent 無法直接與 host 共用網路堆疊

→ 預設封鎖所有出口，僅白名單通行

OpenShell 管控的是「agent 被允許做什麼」（應用層行為）。
但若 agent 利用 kernel exploit 或 container escape 逃出 runc 的 namespace 邊界， 就會直接面對 host kernel，所有 Supervisor 策略失效。
Kata 在此插入一個輕量 VM 邊界；gVisor 在此插入 user-space kernel 過濾。 兩者與 OpenShell 互補，不衝突。

OpenShell 如何「插入」agent-sandbox

OpenShell 不是在 agent-sandbox 外面包一層，而是把 Supervisor binary 注入進 Sandbox CR 的 podTemplate：

OpenShell 建立的 Sandbox CR（節錄）
spec:
  podTemplate:
    spec:
      containers:
        - name: agent
          command: ["/opt/openshell/bin/openshell-sandbox"]  # ← Supervisor，不是 agent 本體
          env:
            - name: OPENSHELL_ENDPOINT
              value: http://openshell.openshell.svc.cluster.local:8080
            - name: OPENSHELL_SANDBOX_COMMAND
              value: sleep infinity                               # ← 真正的 agent 指令
          image: ghcr.io/nvidia/openshell-community/sandboxes/base:latest
      automountServiceAccountToken: false                   # ← 預設關掉 SA token

Supervisor 啟動後才會去執行 OPENSHELL_SANDBOX_COMMAND，並對它套用所有 policy。agent-sandbox controller 對此一無所知，它只看到一個普通 Sandbox CR。

Container Runtime 與 OpenShell 的關係

runtimeClass防禦目標與 OpenShell 的關係本 repo 狀態
runc 無額外隔離，靠 namespace + cgroup 目前使用，需 privileged workaround for netns ✓ 已驗證
Kata Containers 防 container escape、kernel exploit（VM 邊界） 理論上可搭配 OpenShell，但 netns + VM 雙層隔離的相容性未測試 未測試
gVisor 防 syscall 攻擊（user-space kernel） 與 OpenShell 可共存，但目前只驗證到 L7 / policy 層；filesystem policy 退化 已驗證，有限制

💡 Kata / gVisor 解的是「防止 container 逃脫到 host」；OpenShell 解的是「AI agent 的行為治理」。兩者威脅模型不同，可以疊加使用達到縱深防禦。
