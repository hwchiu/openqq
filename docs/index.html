⚠️ k3s + containerd Limitation

OpenShell's agent-sandbox requires privileged: true on the Sandbox CR to create a network namespace. Stock k3s does not set this automatically.

Error without workaround:

mount --make-shared /run/netns
  failed: Permission denied

Fix: k8s/openshell-sandbox-patcher.yaml — a DaemonSet that watches Sandbox CRs and auto-patches privileged: true.

02 — Architecture

OpenShell Component Layers

OpenShell is not just a container — it is a layered runtime. Each layer is independently observable and testable. The pipeline below shows how a request flows from agent to child process.

🖥️ CLI / Agent User or AI agent
issues a command
🌐 Gateway Policy distribution
inference routing
⚙️ K8s Driver Creates
Sandbox CR
🎛️ agent-sandbox CRD controller
spawns Pod
🔒 Supervisor PID 1 in Pod
Landlock + netns
🤖 Child Process Agent runs here
under full policy
Layer 1 Azure IaaS

Virtual Machines, VNet, NSG, public IP — all provisioned by Terraform in rg-azure-k3s-lab-v2.

Terraform Azure eastus Standard_B2s ×3
Layer 2 k3s Cluster

Lightweight Kubernetes distribution. Provides control-plane + worker nodes, CRD support, and containerd runtime.

k3s v1.35.5 containerd 3 nodes
Layer 3 agent-sandbox CRD

Kubernetes SIG project. Defines sandboxes.sandbox.sigs.k8s.io CRD and a controller that reconciles Sandbox CRs into Pods with supervisor injection.

CRD: sandboxes.sandbox.sigs.k8s.io ns: sandbox-system patcher required on k3s
Layer 4 OpenShell Gateway

Central control plane. Distributes policies to sandboxes, routes inference requests to LLM backends, and collects OCSF audit events.

ns: openshell gRPC policy endpoint inference router OCSF aggregator
Layer 5 OpenShell Supervisor

/opt/openshell/bin/openshell-sandbox runs as PID 1. Fetches policy from Gateway, creates a private network namespace, installs Landlock filesystem rules, starts OPA L7 proxy, then exec-s the agent.

Landlock v2 (filesystem) private netns per sandbox OPA engine (L4/L7) TLS termination proxy :3128 hot-reload on policy change

03 — Experiments

Seven Enforcement Experiments

Each experiment targets a specific enforcement layer with real terminal output. Sandbox ID: verify-1780439786 — all seven run inside the same sandbox session.

1

Default Deny Egress

No policy → all outbound network requests blocked
PASS

Before any policy is applied the OPA engine defaults to deny-all. A curl to api.github.com via the HTTPS proxy returns HTTP 403 — proving the network boundary is active before policy loads.

testing/raw/verify-1780439786/default-egress.txt 
$ curl https://api.github.com/zen
curl: (56) CONNECT tunnel failed, response 403

The proxy returns 403 (not a TCP timeout), confirming the Supervisor proxy intercepted and blocked the CONNECT request synchronously.

2

Filesystem Sandbox (Landlock)

/tmp writable · /var/tmp denied · / listing denied
PASS

The Supervisor applies a Landlock ruleset as part of PID 1 startup — before exec-ing the agent process. Rules are static: set at sandbox creation and cannot change without restarting the sandbox.

testing/raw/verify-1780439786/filesystem.txt 
TMP_OK                              # touch /tmp/file → success
touch: cannot touch '/var/tmp/verify-deny': Permission denied
VARTMP_DENIED
testing/raw/verify-1780439786/static-baseline.txt 
uid=998(sandbox) gid=998(sandbox) groups=998(sandbox)
/sandbox
ls: cannot open directory '/': Permission denied

Landlock ruleset: rules_applied:9 skipped:1 — root filesystem browsing is completely blocked.

3

Binary Allowlist (L4 OPA)

curl allowed · python3 denied — same destination, same policy
PASS

The policy includes /usr/bin/curl in the github_api policy's binary allowlist but not python3. OPA evaluates the calling binary at L4, before the connection is opened.

curl → api.github.com (ALLOWED) 
NET:OPEN [INFO] ALLOWED /usr/bin/curl(70) -> api.github.com:443 [policy:github_api engine:opa]
HTTP:GET [INFO] ALLOWED GET http://api.github.com:443/zen [policy:github_api engine:l7]
Avoid administrative distraction.
python3 → api.github.com (DENIED) 
NET:OPEN [MED] DENIED ...python3.14(79) -> api.github.com:443
  [reason:binary '...python3.14' not allowed in any policy]
OSError: Tunnel connection failed: 403 Forbidden
urllib.error.URLError: <urlopen error Tunnel connection failed: 403 Forbidden>
4

L7 Method/Path Control

GET /zen allowed · POST /issues denied — same binary, same host
PASS

After the L4 binary check passes, the L7 TLS-terminating proxy inspects the decrypted HTTP method and path. The policy permits GET /zen but has no rule for POST /repos/.../issues.

POST /repos/octocat/hello-world/issues (DENIED) 
HTTP:POST [MED] DENIED POST http://api.github.com:443/repos/octocat/hello-world/issues
  [policy:github_api engine:l7] [reason:L7_REQUEST deny POST ...]
testing/raw/verify-1780439786/curl-post.txt (structured error) 
{
  "binary": "/usr/bin/curl",
  "error": "policy_denied",
  "detail": "POST /repos/octocat/hello-world/issues not permitted by policy",
  "host": "api.github.com", "layer": "l7", "method": "POST",
  "path": "/repos/octocat/hello-world/issues", "policy": "github_api"
}

The structured JSON error tells the caller exactly which policy and rule caused the denial — far more debuggable than a raw TCP reset.

5

Static vs Dynamic Policy

Filesystem policy is static at creation; network policy hot-reloads
PASS

OpenShell makes a deliberate asymmetry: Landlock rules are applied once and cannot change without a restart. Network (OPA) policy, however, can be pushed live and takes effect in seconds.

Policy hot-reload (from sandbox-logs.txt) 
CONFIG:DETECTED [INFO] Settings poll: config change detected
  [old_revision:9859304338116581045
   new_revision:1086529796914802360
   policy_changed:true provider_env_changed:false]
CONFIG:LOADED  [INFO] Policy reloaded successfully
  [policy_hash:b8cf8d97ded059fe54c8c...]
Pod UID — unchanged before and after policy reload 
Pod UID before:  f9698053-7ec0-4656-a7c7-8c025e66ebdc
Pod UID after:   f9698053-7ec0-4656-a7c7-8c025e66ebdc  ← same pod
6

ServiceAccount Token Isolation

K8s SA token absent · OpenShell bootstrap token present
PASS

A compromised agent should not be able to pivot to the Kubernetes API. OpenShell does not mount the default ServiceAccount token inside the sandbox.

K8S_SA_ABSENT — no /var/run/secrets/kubernetes.io/serviceaccount/token found inside the sandbox. The agent cannot authenticate to the k3s API server. OPENSHELL_BOOTSTRAP_PRESENT — the scoped bootstrap token allows Supervisor ↔ Gateway communication only.

This prevents a class of supply-chain attacks where a malicious dependency reads the SA token and exfiltrates cluster access credentials.

7

Containerd Workaround — Auto-Patcher

Sandbox CR auto-patched with privileged: true on k3s
PASS

The agent-sandbox controller requires privileged: true on the Sandbox CR to create a private network namespace. Standard k3s does not grant this, so every sandbox creation fails.

Workaround: k8s/openshell-sandbox-patcher.yaml deploys a DaemonSet that watches for new Sandbox CRs and immediately patches them with the required security context — no manual intervention needed per sandbox.

Result after patch

06 — Reproduce

How to Reproduce

The entire lab can be rebuilt from scratch in under 15 minutes. You need an Azure subscription, Terraform ≥ 1.9, and kubectl.

1 — Clone and configure

git clone https://github.com/hwchiu/openqq.git && cd openqq
cp terraform/terraform.tfvars.example terraform/terraform.tfvars
$EDITOR terraform/terraform.tfvars  # fill in Azure credentials

2 — Provision the cluster

make tf-init && make tf-plan  # review resources
make cluster-up              # apply + wait ~5 min
make cluster-kubeconfig      # writes generated/kubeconfig
kubectl --kubeconfig generated/kubeconfig get nodes

3 — Deploy OpenShell + agent-sandbox

kubectl apply -f k8s/agent-sandbox/  # CRD + controller
kubectl apply -f k8s/openshell/       # Gateway + Driver
bash scripts/install-openshell-sandbox-patcher.sh

4 — Run validation

bash scripts/run-validation.sh
All 7 experiments passed ✓
# Evidence written to testing/raw/verify-$(timestamp)/

5 — Tear down

make cluster-down  # terraform destroy — removes all Azure resources
