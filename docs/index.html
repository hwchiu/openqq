openqq、agent-sandbox、OpenShell 各自負責什麼？

這三個名詞很容易混在一起，但它們分屬不同層次，負責的事情完全不同。

🧪
openqq（本專案）

黑箱測試框架，不是沙箱本身。負責：

  • 用 Terraform 在 Azure 建立 k3s 叢集
  • 安裝 OpenShell stack
  • 從外部執行驗證腳本（curl、python3…）
  • 收集與保存原始測試輸出

📌 openqq 自己不提供任何沙箱或安全能力。

☸️
agent-sandbox（k8s SIG）

Kubernetes SIG 專案，提供 Sandbox CRD 與 controller。負責：

  • 為每個 Sandbox CR 建立 Pod / Service / PVC
  • 維持 stable identity 與 lifecycle
  • 保證 singleton 語意

📌 agent-sandbox 是底層承載器，不負責 policy enforcement 或安全控制。

🔒
OpenShell（NVIDIA）

完整的 AI Agent 安全 runtime 系統。負責：

  • CLI：操作入口（建立 sandbox、套用 policy）
  • Gateway：control plane，管理政策版本與 session
  • Driver：將抽象語意轉成 k8s / 其他平台操作
  • Supervisor：在容器內執行所有 enforcement

📌 真正的安全特性（Landlock、netns、L7 policy、熱重載）全部由 OpenShell 提供。

三者的關係

OpenShell 利用 agent-sandbox 來承載 workload；agent-sandbox 只管 Pod 生命週期，安全控制全由 OpenShell 的 Driver + Supervisor 負責。openqq 則站在系統外部，驗證整個堆疊的行為是否符合預期。

agent-sandbox 的底層 Runtime：Kata / gVisor 與 OpenShell 的關係

agent-sandbox 本身只是一個 CRD，它建出來的 Pod 可以套用不同的 runtimeClassName， 選擇不同的 container runtime 來獲得更強的底層隔離。常見選項包括：

runc（預設）

標準 container runtime，共享 host kernel。本 repo 目前使用此模式。

隔離靠 namespace + cgroup，無額外 kernel 隔離。

Kata Containers

每個 Pod 跑在獨立輕量 VM（QEMU / Firecracker）。提供完整 kernel 邊界隔離。

防禦目標：container escape、kernel exploit。

gVisor（runsc）

User-space kernel，以 gofer 攔截所有 syscall。減少攻擊面但不是完整 VM。

防禦目標：syscall 攻擊、host kernel 曝露。

功能重疊分析

Kata / gVisor 與 OpenShell 在部分領域看似做了類似的事，但實際上解決的是不同層次的問題：

能力Kata ContainersgVisorOpenShell Supervisor層次
Kernel 邊界隔離 ✓ VM 隔離 ✓ Syscall 攔截 ✗ 基礎設施層
Filesystem 存取限制 ✓ VM 邊界 ✓ Syscall 層 ✓ Landlock LSM（細粒度） ⚠ 重疊，但粒度不同
Network namespace 隔離 ✓ 獨立 VMM 網路 ✓ 獨立 network stack ✓ veth + netns + proxy ⚠ 重疊，但機制不同
出口流量 L7 政策（HTTP method/path） ✗ 無感知 ✗ 無感知 ✓ Supervisor Proxy OpenShell 獨有
Binary 級別 allowlist ✗ ✗ ✓ Supervisor OpenShell 獨有
Policy 熱重載（不重啟 Pod） ✗ ✗ ✓ Gateway + Supervisor OpenShell 獨有
OCSF 結構化安全日誌 ✗ ✗ ✓ Supervisor OpenShell 獨有
防止 container escape / kernel exploit ✓ 最強 ✓ 部分 ✗ 非設計目標 基礎設施層

結論：不是替代，是互補

Kata / gVisor 解的是「如何防止 container 逃脫到 host」（基礎設施層安全）。
OpenShell 解的是「允許 AI Agent 做哪些事」（應用層政策管控）。
兩者可以同時使用，形成縱深防禦（Defense in Depth）： Kata 防止 kernel-level 攻擊，OpenShell 控制 agent 行為。

⚠️ 注意：本 repo 目前使用標準 runc（k3s 預設）， 尚未測試 Kata / gVisor 與 OpenShell Supervisor 共存時的相容性。 已知 k3s + containerd + netns 已有限制需要 privileged workaround， 加上 Kata 可能有額外的配置需求。

7 項驗證實驗

01
預設出口封鎖Network
✓ 通過

在尚未套用任何政策的情況下，所有出口連線應被封鎖。

default-egress.txt
$ curl -v --max-time 10 https://api.github.com/zen
curl: (56) CONNECT tunnel failed, response 403

HTTP CONNECT 通道被 Proxy 以 403 拒絕，出口封鎖確認有效。

02
Landlock 檔案系統沙箱Filesystem
✓ 通過

驗證 Landlock 正確允許 /tmp 寫入，並拒絕 /var/tmp 及根目錄列舉。

filesystem.txt
TMP_OK
touch: cannot touch '/var/tmp/verify-deny': Permission denied
VARTMP_DENIED
static-baseline.txt
ls: cannot open directory '/': Permission denied

Landlock ABI v4 可用，實際套用 V2，規則 9 條，跳過 1 條（cgroup 不可用）。

03
政策放行 — curl GETL7 Policy
✓ 通過

設定政策允許 api.github.com 的 HTTPS GET，驗證放行後可正常取得回應。

curl-get.txt
Avoid administrative distraction.

GitHub API /zen 端點回傳格言，確認 GET 請求通過。

04
政策封鎖 — python3 GETL7 Policy
✓ 通過

政策僅允許 curl agent，python3 的 HTTP 請求應被 L7 層拒絕。

python-get.txt
urllib.error.URLError: <urlopen error>
  <HTTP Error 403: Forbidden>

Proxy 回傳 403 Forbidden，python3 請求被 L7 政策封鎖。

05
政策封鎖 — curl POSTL7 Policy
✓ 通過

即使 GET 被放行，POST 方法應被 L7 政策拒絕並回傳結構化錯誤。

curl-post.txt
{
  "error": "policy_denied",
  "layer": "l7",
  "method": "POST",
  ...
}

Supervisor 回傳結構化 JSON 拒絕訊息，HTTP 方法層級的管控正常運作。

06
政策熱重載Lifecycle
✓ 通過

更新政策後，Supervisor 應在不重啟 Pod 的情況下套用新政策，Pod UID 應維持不變。

pod_uid_before.txt / pod_uid_after.txt
f9698053-7ec0-4656-a7c7-8c025e66ebdc  ← 更新前
f9698053-7ec0-4656-a7c7-8c025e66ebdc  ← 更新後（相同）

兩次查詢 Pod UID 完全相同，確認熱重載不會造成 Pod 重啟。

07
k3s privileged 工作繞過Platform
✓ 通過

k3s + containerd 不允許在 Sandbox CR 建立 netns，需自動加上 privileged: true。

錯誤原因
mount --make-shared /run/netns failed: Permission denied
解決方式
$ kubectl apply -f k8s/openshell-sandbox-patcher.yaml
daemonset.apps/sandbox-patcher created
# DaemonSet 監聽新建的 Sandbox CR，自動 patch privileged: true

部署 DaemonSet patcher 後，所有新 Sandbox CR 自動取得必要權限。

如何重現這個實驗

以下步驟假設你已安裝 Terraform、Azure CLI 並已登入 Azure 帳號。

1
Clone 並設定環境 
$ git clone https://github.com/hwchiu/openqq.git && cd openqq
$ cp terraform/terraform.tfvars.example terraform/terraform.tfvars
# 編輯 terraform.tfvars 填入 Azure subscription_id
2
建立 Azure 基礎設施 
$ make infra-up
# 約 3–5 分鐘，建立 VM、VNet、NSG
3
安裝 k3s 叢集與 OpenShell 
$ make k3s-install
$ make openshell-install
$ bash scripts/install-openshell-sandbox-patcher.sh
# patcher 解決 k3s + containerd 的 netns 問題
4
執行完整驗證 
$ make verify
# 所有 7 項實驗輸出儲存至 testing/raw/verify-<timestamp>/
5
清理資源 
$ make infra-down
# 銷毀所有 Azure 資源，避免計費
