System Split

先把三個維度拆開

整個系統最容易誤解的地方，是把 `agent-sandbox`、`OpenShell`、`runtime` 混成同一件事。實際上它們各自解不同問題，而且可以自由組合。

01

Workload Lifecycle

agent-sandbox 是 Kubernetes 抽象層，負責 Pod / Service / PVC / 單例語意與生命週期。它本身不是安全產品。

代表物件：`Sandbox` CRD
02

Runtime Boundary

runc / gVisor / Kata 決定容器和 kernel 的邊界強度。這層保護的是「逃逸到 host」的難度，不是 agent 可以呼叫哪些 API。

代表物件：`runtimeClassName`
03

Behavior Governance

OpenShell Supervisor 決定 agent 在 sandbox 內能做什麼：L7、binary allowlist、filesystem policy、hot-reload、OCSF audit。

代表物件：Gateway / Supervisor / policy
Runtime Verdict

三條實驗線，不講空話

每一條 runtime 都用相同問題去驗證：OpenShell sandbox 能不能起、L7 還在不在、filesystem policy 還在不在、阻塞點到底卡在哪一層。

runc Recommended Baseline

完整保留 OpenShell 的 filesystem、network、L7、policy hot-reload。這是目前唯一已證實功能全開的主線。

  • Landlock 成立
  • L7 成立
  • Azure File CSI 可掛入 Sandbox
查看完整實驗
gVisor Partial Retention

runtime 邊界更強，但 `landlock_create_ruleset` 在這條路徑回 `ENOSYS`，所以 filesystem policy 退化。

  • L7 仍成立
  • hot-reload 仍成立
  • filesystem policy 失效
查看 gVisor 報告
Kata Blocked by Privileged

VM 邊界最強，但 OpenShell sandbox 目前需要的 privileged bootstrap 直接撞上 Kata 的相容性限制。

  • `/dev/kvm` 已驗證
  • guest kernel 已驗證
  • OpenShell sandbox 尚未打通
查看 Kata 報告
Core Idea

OpenShell 真正補的是什麼

如果你不用 Kata / gVisor，OpenShell 也不是沒意義。它補的是 agent 行為治理，不是 VM 邊界。這兩者互補，但不能混為一談。

Host Layer

Linux Kernel + containerd

runc 路徑仍共享 host kernel。若發生 kernel exploit，OpenShell 的應用層政策不會自動變成 VM 邊界。

Container Layer

Sandbox Pod / runtimeClass

這層決定「容器怎麼執行」：runc 是 baseline，gVisor 是 user-space kernel，Kata 是輕量 VM。
