系統架構

現在這個 lab 的實際鏈路不是單一元件，而是由 control plane、compute driver、Kubernetes 承載層與 sandbox 本地 enforcement 一起組成。

CLI 建立 sandbox、連線、套 policy、讀 logs
Gateway 管理 state、policy、provider、settings、relay
K8s Driver 把抽象 sandbox 轉成 Kubernetes `Sandbox` 物件
agent-sandbox reconcile CRD，建立 Pod / Service / PVC
Supervisor 套 process / filesystem / network enforcement

核心概念

OpenShell

整個系統名稱，不是單一 Pod。包含 CLI、Gateway、Supervisor、policy 模型與 compute driver。

OpenShell Sandbox

被 OpenShell runtime 保護的執行環境。它表面上像 Pod，但實際上有自己的本地 enforcement。

agent-sandbox

Kubernetes SIG 專案，提供 `Sandbox` CRD/controller。它是承載層，不是 OpenShell policy engine。

Driver 是什麼

Driver 負責把 OpenShell 的 sandbox 抽象語意翻譯成平台動作。現在這個環境用的是 Kubernetes driver，證據是 gateway config 與 runtime logs 都出現 openshell.drivers.kubernetes。

Supervisor 是什麼

Supervisor 是 sandbox 內真正的 enforcement 元件。它建立 network namespace、套 Landlock、處理 egress proxy、載入 policy，最後才啟動 child process。

已完成實驗

實驗 要證明什麼 結果 意義
Default deny egress OpenShell 不是單靠 K8s 網路，sandbox 預設就拒絕未授權外連 已驗證 curl https://api.github.com/zen 預設回 403
Filesystem sandbox child process 的可讀寫路徑受本地 policy 約束 已驗證 /tmp 可寫，/var/tmp 不可寫，ls / 被拒絕
Binary allowlist 同一 endpoint 可依 binary 身份決策 已驗證 curl 可連 GitHub，python3 不可連
L7 method/path control OpenShell 可做到 HTTP method/path 級別控制 已驗證 GET /zen 允許，POST /repos/.../issues 拒絕
Static vs Dynamic policy filesystem/process 與 network policy 並不是同一層 已驗證 live update 後 GitHub 規則立即生效，但 /var/tmp 仍不能寫
ServiceAccount exposure sandbox child process 沒有直接拿到標準 K8s SA token 已驗證 K8S_SA_ABSENT，但 OPENSHELL_BOOTSTRAP_PRESENT
Containerd workaround automation 目前 k3s + containerd 缺的 privileged workaround 已能自動化 已驗證 新建 sandbox 不需手動 patch 也可進入 Ready

Runtime 與後續路線

目前狀態

  • Azure 三節點 k3s 叢集已驗證
  • 底層 runtime 是 containerd
  • OpenShell 與 agent-sandbox 已運作
  • OpenShell sandbox 需透過 patcher 自動補上 privileged workaround

下一步

  • Terraform 已新增 container_runtime 與 crio_version
  • cloud-init 已具備安裝 CRI-O 與把 K3s 指向 CRI socket 的邏輯
  • 下一輪可直接切到 container_runtime = "crio" 重建叢集
  • 之後重跑同一套 OpenShell 驗證腳本，比較 containerd / CRI-O 差異
