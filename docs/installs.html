INSTALL PATHS

Independent environment lines

These are the four supported roots currently surfaced in the docs and matrix JSON.

Path 01

K3s + gVisor

Pure runtime-isolation baseline with no extra security control plane layered on top.

Install command
./scripts/install-k3s-gvisor.sh
Open runbook
Path 02

K3s + OpenShell + runc

Primary full-capability path with verified filesystem controls, L7 policy, hot-reload, and CSI evidence.

Install command
./scripts/install-k3s-openshell-runc.sh
Open runbook
Path 03

K3s + OpenShell + gVisor

OpenShell control plane over a gVisor runtime boundary. L7 remains, filesystem policy degrades.

Install command
./scripts/install-k3s-openshell-gvisor.sh
Open runbook
Path 04

K3s + KubeArmor + runc

Independent runtime-security comparison line for KubeArmor evidence and SA-token protection checks.

Install command
./scripts/install-k3s-kubearmor-runc.sh
Open runbook
MATRIX WORKFLOW

Provision, validate, publish, destroy

The matrix scripts exist so you can build all four environments with shared naming and then run one published comparison pass against them.

Standard sequence
./scripts/install-comparison-matrix.sh
PUBLISH_RESULTS=true ./scripts/run-comparison-matrix-tests.sh
./scripts/destroy-comparison-matrix.sh
  • The install script keeps each stack isolated with separate Terraform state and kubeconfig.
  • The runner executes the shared test catalog and writes evidence into `testing/results/latest` plus the matrix data snapshot.
  • The destroy script tears the matrix back down after the comparison window closes.
GUIDANCE

Which path to start with

Need the full OpenShell feature set

Start with `k3s-openshell-runc`.

Need the gVisor baseline only

Start with `k3s-gvisor` and verify the runtime boundary first.

Need published four-stack evidence

Use the matrix workflow rather than a single install path.
