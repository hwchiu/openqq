2026-06-08 Live Rerun & Hardening Updates

2026-06-08 重新建立四套環境後執行 live matrix rerun（含新增 Istio 疊加測試），並對 KubeArmor 進行 root cause 修正與 agentic scenario 擴充驗證。

KubeArmor Hardening (06-08)
  • Root cause: install timing + annotateExisting=false 導致既有 workload 未被正確處理。
  • 修正: install script 加入 --wait + annotateExisting=true，調整 workload 建立順序。
  • 結果: clean rebuild 後 kubearmor-sa-block 從 FAIL → PASS。
  • 證據: kubearmor-hardening-2026-06-08.md
Agentic Scenarios (KubeArmor)
  • File/secret guardrails 穩定（PASS）。
  • Process / network / shell escape / egress 多數 FAIL 或 DEGRADED。
  • AppArmor profile 已包含規則，但 runtime 行為仍有差距。
  • 適合保護 secret/file read，不適合全面防 agent self-extension。
Istio 疊加影響
  • control-plane & sidecar-smoke: 四套全部 PASS。
  • gVisor sidecar: FAIL（iptables nat 無法初始化 in guest kernel）。
  • OpenShell + runc / KubeArmor + runc 的 guardrails 在 Istio 後仍維持 PASS。
  • 詳細: istio-impact-2026-06-08.md

完整 live 報告：comparison-matrix-live-2026-06-08.md + KubeArmor 擴充 kubearmor-agentic-scenarios-2026-06-08.md。

02

架構模型

四套環境共享以下基礎層：

  • Azure VM + Terraform 管理
  • k3s + containerd
  • agent-sandbox CRD + controller（負責 Pod / Service / PVC 承載）

差異發生在「執行環境治理層」：

OpenShell 路線
Gateway + Supervisor 模型
• Gateway（openshell-0）負責 control plane 與 policy 版本控管
• Supervisor 跑在每個 sandbox Pod 內，負責本地 Landlock、netns proxy、binary allowlist、L7 過濾
• sandbox 與 gateway 維持長連線 session
gVisor 路線
Userspace Kernel 邊界
• runsc 提供獨立 guest kernel
• syscall 層隔離，但 Landlock 等 Linux 安全機制在目前版本不可用
• OpenShell 的 network/L7 能力仍可運作

實驗過程與可重現性

統一建置
./scripts/install-comparison-matrix.sh

一次建立四套獨立環境（各自的 Terraform state、VNet、kubeconfig）。

測試與發佈
PUBLISH_RESULTS=true ./scripts/run-comparison-matrix-tests.sh

使用 testing/matrix/catalog.json 驅動，結果發佈至 docs/data/comparison-matrix.json。
