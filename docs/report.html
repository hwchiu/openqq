Executive Report / Platform Decision

別再問哪個 runtime 比較帥

這頁只回答平台決策。目標不是選一個名字最安全的 runtime，而是判斷 OpenShell 的哪些能力對你重要，以及這些能力在不同 runtime 上是不是還存在。

看 runc 基準組 看 raw evidence

最短結論

現在能正式落地的是 runc + OpenShell。 gVisor 是明確的隔離支線，但會失去 filesystem policy。Kata 目前還是 blocker analysis，不是部署路線。

判斷基準

  • L7 / binary allowlist 是否保留
  • filesystem policy 是否保留
  • runtime 邊界是否提升

Verdict

最終判讀

不要把這個 repo 的結論讀成「gVisor 比 runc 安全」或「Kata 比 gVisor 更高級」。這些說法都太粗。真正要看的，是每條路徑上 OpenShell 的治理能力還剩多少。

Recommended

runc + OpenShell

這是目前功能最完整的主線。filesystem、network、L7、binary allowlist、hot-reload 都在。

Trade-off

gVisor + OpenShell

更強的 runtime 邊界，換來 Landlock 缺口。L7 和 hot-reload 仍成立，但 filesystem policy 失效。

Blocked

Kata + OpenShell

VM 邊界最強，但目前不是產品化方案，因為 sandbox 還起不來。這頁的價值在 blocker 定位，不在功能展示。
