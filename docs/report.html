一句話結論

推薦路徑

如果目標是讓 OpenShell 功能完整落地，現在最穩定的選項仍是 runc + OpenShell。如果目標是提高 runtime / kernel 邊界，gVisor + OpenShell 已可運作，但目前只能保留 network / L7 / hot-reload，不能保留 OpenShell 的 filesystem policy。Kata + OpenShell 則已確認 runtime 可用，但 sandbox 被 `privileged` 相容性阻塞。

不該怎麼解讀

這批結果不支持「gVisor 全面優於 runc」這種說法。它支持的是：gVisor 與 runc 代表兩種不同的安全取向，而 OpenShell 在兩條路徑上保留下來的能力不同。

已確認的根因

🧪
對照實驗

在同一個叢集建立兩個最小 Pod，一個走 `runc`，一個走 `runtimeClassName: gvisor`，直接呼叫 `landlock_create_ruleset` syscall。

📌
實驗結果

`runc` 回 `ret=4`，代表 ABI v4 可用；`gVisor` 回 `ENOSYS`，代表該 syscall 在 gVisor 路徑未實作。這就是 OpenShell filesystem policy 退化的直接原因。

Root Cause Snapshot
runc:   ret=4   errno=0   errno_name=NONE
gvisor: ret=-1  errno=38  errno_name=ENOSYS

建議採用策略

Option A

短期上線

以 `runc + OpenShell` 作為主要落地方案。這是目前唯一已驗證同時保有 filesystem、network、L7 與 hot-reload 的組合。

Option B

隔離強化支線

保留 `gVisor + OpenShell` 作為強 runtime 邊界的支線，用於不依賴 filesystem policy 的 agent 類場景。

Option C

第三條線的現況

`Kata Containers + OpenShell` 已不是待評估狀態，而是已完成第一輪相容性定位：runtime 可用，但 OpenShell sandbox 被 `privileged` 相容性阻塞。

替代方向

方向目前判讀理由
`server.enableUserNamespaces=true`不是直接解法官方文件把它定位成 defense-in-depth；supervisor 啟動順序仍保留 privileged bootstrap helpers 與 netns setup。
`supervisor.sideloadMethod=init-container`不是主因我們的最小矩陣已證明 `image volume` 可在 Kata 下單獨運作；真正阻塞的是 `privileged`。
OpenShell `vm` compute driver最值得追官方本來就提供 VM-backed sandbox 路徑，目標與「想要 VM 邊界」完全一致。
