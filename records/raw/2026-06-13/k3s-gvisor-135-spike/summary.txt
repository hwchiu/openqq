K3s gVisor 1.35 spike summary
Date: 2026-06-13
Stack: k3s-gvisor-135
Versions:
- K3s: v1.35.5+k3s1
- CRI-O: 1.35.4
- gVisor: runsc release-20260608.0

Observed live results:
- Terraform apply succeeded and all 3 nodes registered.
- Cilium installed successfully and all 3 nodes reached Ready.
- Current repo gVisor wiring (runtime_type=oci, runtime_path=/usr/bin/runsc) allowed verify pod sandbox creation, image pull, container creation, and StartContainer.
- Short-lived probe pod entered Error with exit code -1 / bogus finished timestamp in Kubernetes view.
- CRI-O journal repeatedly logged: Failed to find container exit file.
- runsc list showed the verify container and sandbox stopped, proving the workload actually ran and exited.
- A long-running sleep pod improved one step further: runsc list showed sandbox/container running, but CRI-O still reported the container as CONTAINER_CREATED and Kubernetes kept the pod in ContainerCreating.
- A more upstream-style vm+shim trial on stock CRI-O 1.35.4 exposed a first hard gate:
  - CRI-O warning: Runtime handler "runsc" is being ignored due to: invalid runtime_path for runtime 'runsc': containerd binary naming pattern is not followed
  - kubelet error: failed to find runtime handler runsc from runtime list map[crun:... runc:...]
- A naming-gate workaround using /usr/local/bin/containerd-shim-runsc-v2 -> /usr/bin/containerd-shim-runsc-v1 moved the failure one step deeper:
  - runsc handler became registered
  - pod sandbox failed with: dial unix \\0{\"version\":2,\"address\":\"unix:///run/containerd/s/...\",\"protocol\":\"ttrpc\"}: connect: invalid argument
  - this indicates stock CRI-O 1.35.4 still treated the shim bootstrap JSON as a raw dial target instead of parsing the address field

Interpretation:
- 1.35 is materially better than 1.34 for the repo's current oci/runsc path because workloads now start and run under runsc.
- It is still not a PASS for Kubernetes compatibility because CRI-O 1.35 does not correctly surface runsc container lifecycle state to kubelet in this setup.
- The vm+shim path is not a viable fix on stock CRI-O 1.35.4 in the current environment.
- The live evidence now points to two distinct stock-CRI-O gaps: runtime path naming acceptance and shim bootstrap JSON parsing.
